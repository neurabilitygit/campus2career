Rising Senior CTO technical brief

An evidence-driven career readiness platform for students, parents, and advisors.

Rising Senior is organized as a TypeScript monorepo with a Next.js web application, a Node API, a background worker, shared scoring contracts, and PostgreSQL persistence. The technical goal is to convert fragmented student, academic, document, and labor-market data into explainable readiness scores and practical guidance.

System goal

The platform evaluates whether a student's current college path is producing credible evidence for a desired job after graduation. It combines student profile data, curriculum requirements, transcript and artifact extraction, occupation data, market signals, role skills, deterministic scoring, and LLM-generated narrative guidance.

Application organization

Web app apps/web is a Next.js application. It handles Supabase OAuth session state, onboarding, document upload, dashboards, and role-context navigation.
API service apps/api exposes authenticated student, parent, coach, academic, market, upload, scoring, brief, and chat routes.
Worker apps/worker polls queued parse jobs, downloads uploaded artifacts, extracts text, updates artifact state, and performs transcript/catalog follow-up work.
Shared contracts packages/shared centralizes scoring types, career contracts, target role seeds, and cross-app payload shapes.

Primary data flow

  • User signs in through Supabase OAuth; the API verifies Supabase JWTs and resolves the application role and student context.
  • Student onboarding stores school, major, minor, geography, sector selections, and target job data in PostgreSQL.
  • Academic directory and catalog routes load institution, program, major, minor, and requirement structures from seeded data, official pages, uploaded PDFs, and LLM-assisted extraction when appropriate.
  • Upload routes mint Supabase Storage signed upload URLs, verify ownership and object existence, persist artifacts idempotently, and queue parse jobs.
  • The worker consumes parse jobs and updates artifact, transcript, and catalog-derived evidence records.
  • The scoring engine aggregates role requirements, market signals, academic evidence, artifacts, contacts, outreach, deadlines, and execution signals into readiness outputs.
  • Parent brief and scenario-chat services convert scoring and context into user-facing narrative guidance, with fallbacks when LLM delivery is unavailable or too slow.

Use of LLMs

GPT-5.4 is used for structured scenario guidance, parent brief generation, job-target normalization, and LLM-assisted coursework extraction from official school-page snippets. The system keeps deterministic scoring separate from LLM narrative generation so user-facing advice can be improved without making the core score opaque.

The current hardening work adds SDK-level timeout cancellation for OpenAI calls, structured fallback behavior, and telemetry through llm_runs so provider failures, timeouts, and degraded responses can be inspected.

Key data sources

  • Student-entered profile, target job, geography, and sector preferences.
  • Official or seeded college institution and catalog data.
  • Uploaded transcripts, resumes, program requirement PDFs, and supporting artifacts.
  • O*NET-derived occupation mappings and role-skill requirements.
  • BLS and seeded market signals used by the market-demand scoring layer.
  • LLM-assisted extraction outputs that are labeled separately from direct-source or uploaded-PDF evidence.

Expected technical benefits

  • Single source of truth for student readiness across web, API, worker, and shared scoring contracts.
  • Traceable score inputs and outputs, including market metadata, skill gaps, academic progress, and next actions.
  • Safer upload processing through server-minted paths, ownership checks, storage existence checks, idempotent persistence, and worker state transitions.
  • Better operational observability through explicit LLM run status, parse-job status, and route-level error responses.
  • Extension path for headless APIs, job-platform integrations, richer school catalog ingestion, and partner data feeds.

Current review priorities

  • Continue validating real parent-student household linking with separate accounts, not only test-context switching.
  • Improve target-goal UX so free-text career goals cannot silently disagree with structured scored targets.
  • Expand seeded institution, catalog, role, market, and requirement coverage for faster first-run value.
  • Keep LLM output visibly labeled when it is inferred or fallback-generated rather than sourced directly.
  • Finalize staging infrastructure, secrets management, CI checks, and production-safe observability before external launch.
